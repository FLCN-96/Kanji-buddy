# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Running locally

There is **no build step, no package.json, no test suite, and no linter**. React 18 + Babel-standalone are loaded from a CDN and JSX is transpiled in the browser.

Serve the directory with any static server and open `Home.html`:

    python -m http.server 8000
    # → http://localhost:8000/Home.html

Pushes to `main` deploy to GitHub Pages via `.github/workflows/main.yml` (uploads the repo root as the artifact — there is no build step in CI either).

PNG icons are generated by opening `tools/gen-icons.html` in a browser and downloading each canvas into `icons/`. The SVG sources are checked in; the PNG outputs are not regenerated automatically.

## Architecture

### Multi-page shell, shared globals

Each mode is a standalone HTML page (`Home.html`, `Run.html`, `TimeAttack.html`, `Survival.html`, `StreakGuard.html`, `LeechHunt.html`, `Match.html`, `Settings.html`) that:

1. Loads the shared data-layer scripts in fixed order: `data/rank.js` → `data/db.js` → `data/load-cards.js` → `data/daily.js` → (`data/srs.js` for Run only).
2. Loads React/ReactDOM/Babel from unpkg.
3. Loads the page's `.jsx` components as `type="text/babel"`.
4. Calls `await window.loadCards()` and renders the page's top-level `<XxxApp cards={cards} />` into `#root`.
5. Registers `sw.js`.

The data-layer scripts are plain IIFEs that attach singletons to `window` — `window.DB`, `window.Daily`, `window.Srs`, `window.Rank`, `window.loadCards`. Components consume them off `window` directly; there are no imports.

When adding a new page or component, mirror this pattern: add the file to the page's HTML script tags **and** to the `LOCAL_ASSETS` list in `sw.js` (otherwise it won't be precached or available offline).

### Persistence (`data/db.js`)

IndexedDB database `kanji-buddy-db`, current version **3**, four object stores:

- `user_profile` (single record at `id=1`) — display name, total_xp, streak counters, settings blob.
- `card_states` (keyPath `idx`, indexed by `due_date`) — SM-2 state per card: `interval_days`, `ease_factor`, `reviews`, `lapses`, `due_date`, `last_reviewed`.
- `sessions` (autoIncrement) — one row per completed run/challenge.
- `scores` (autoIncrement, `mode_score` index) — best-of tracking for challenge modes.

If you need to add a store or index, bump `DB_VERSION` and extend the `onupgradeneeded` block. Existing user data must survive — only delete stores that are obsolete (the `imported_cards` removal is the existing precedent).

`DB.grantXp(amount)` is the **only** sanctioned way to award XP. It updates `total_xp` and calls `Rank.flagPromotion(prev, total)`, which writes `kb-promotion-pending` to localStorage. `Home.html`'s `App` consumes that key on mount and triggers `RankUpModal`. Modes that award XP must use `grantXp` so promotions fire correctly.

### Daily deck (`data/daily.js`)

`Daily.selectDailyDeck(cards, cardStates)` builds the **5-card** Run deck with budgets `2 new + 2 due + 1 leech`, with cascade-fill if any bucket is short. A card is a "leech" at `lapses ≥ 3`. Cards are returned augmented with `_bucket` (`'new' | 'due' | 'leech'`) and `_state` (the existing card_state, or null for new). New cards always sort first so the intro/learn phase precedes the quiz.

Home gates the panel as **clear** once `reviewed-today >= Daily.DECK_SIZE`, even if the SRS backlog is non-zero — otherwise the ~2000-card "new" pool would refill the panel forever. The raw SRS backlog is shown separately in `StatusBar`. See `components/App.jsx` lines ~146–175.

`Daily.daySeed()` and `Daily.hotChallengeId()` are deterministic per local day. The "hot" challenge gets a `Daily.HOT_MULTIPLIER` (3×) XP bonus — challenge modes apply this themselves at end-of-session.

### SRS (`data/srs.js`)

Standard SM-2. Verdict → quality: `miss=1, hard=3, ok=4, easy=5`. Lapse path (`q<3`) resets `interval_days=0`, `reviews=0`, increments `lapses`, and reschedules `+6h` for same-day relearn. Pass path uses fixed graduation (1d, 6d) for the first two reviews, then `interval × ease`. Easy adds a 1.3× bonus. Ease floor 1.3.

Run is the only consumer; challenge modes do not touch card_states.

### Rank ladder (`data/rank.js`)

12 ranks (`Ⅰ NOVICE` → `Ⅻ ASCENDANT`), thresholds 0 → 55,000 XP. Four colour tiers: cyan / magenta / amber / transcend. `Rank.flagPromotion(prev, total)` writes a one-shot localStorage marker; `Rank.consumePromotion()` reads-and-clears it.

Rank thresholds are deliberate — don't tweak them without thinking about how they interact with the per-mode XP balances (`RUN_XP`, `TA_XP_*`, etc., defined at the top of each mode's component).

### Cards (`data/cards.json`)

Large (≈6.7 MB) bundled deck. Each card has `idx` (1-based, used as the card_state key), `k` (kanji), `jlpt`, `on`/`kun` readings, `mean`, examples, `svg` (KanjiVG stroke order), `strokes`. The order is roughly frequency / JLPT — `selectDailyDeck` relies on this for stable new-card priority.

### Settings & visual variants

`localStorage['kb-tweaks']` stores `{ variant, accent, scanlines, density, hero }`. `App.jsx` writes them to `document.body.dataset.*`, and CSS keys off those data attributes. The current shipped variant is `game` (locked in by 74a8404). When adding a tweak, add it to `TWEAK_DEFAULTS` in **both** `App.jsx` and `Settings.jsx`, and in every mode's `TWEAK_DEFAULTS_*`.

### Service worker (`sw.js`)

`kb-v9` cache. Source files (`.jsx`/`.css`/`.html`) use stale-while-revalidate so deploys land on the next visit without bumping the cache version. Heavy assets (`.json`/`.svg`/`.png`/`.woff2`) are cache-first. Bump `CACHE` to `kb-v10` only when you need to force eviction (e.g. removed assets or changed precache list).

## Conventions

- All code is written for the browser. There is no Node, no bundler, no transpiler outside Babel-standalone — keep components in plain JSX with `React.useState` / `React.useEffect` (no hook imports).
- Components don't import each other. They attach to `window` via `Object.assign(window, { ... })` at the bottom of the file and reference siblings as bare globals (`<RankUpModal ... />`, `window.Daily.X`).
- Per-mode CSS lives in a single top-level `<mode>.css` file. Shared design tokens are in `design_system/colors_and_type.css`.
- Data attribute hooks: `data-screen-label` on `<main>` for analytics-friendly screen IDs; `data-phase` on shells for phase-scoped CSS.
- `.claude/` is gitignored — it contains host-specific paths and must not be committed.
